# 帧同步 vs 状态同步 (Frame Sync vs State Sync)

## 核心区别

| 特性 | 帧同步 (Lockstep / Deterministic Lockstep) | 状态同步 (State Synchronization) |
| :--- | :--- | :--- |
| **同步内容** | **玩家的操作指令** (Input) | **游戏实体的状态** (Position, HP, State) |
| **计算位置** | **客户端计算** (Client-side Simulation) | **服务器计算** (Server-side Authoritative) |
| **确定性** | **严格要求确定性** (Deterministic) | 不要求严格确定性 (Non-deterministic) |
| **流量消耗** | 极低 (仅同步按键)，随玩家数线性增长 | 较高 (同步所有变化的状态)，随物体数和玩家数增长 |
| **反作弊** | 较难 (需要全盘校验或录像回放)，易出现透视挂、加速挂 | 较好 (服务器权威，客户端只负责表现)，主要防瞬移、修改数值 |
| **断线重连** | **慢** (需要从初始状态重跑所有指令，或定期保存快照) | **快** (直接下发当前最新状态) |
| **适用类型** | RTS (星际、魔兽)、格斗游戏、局域网联机 | MMORPG (魔兽世界)、FPS (CS:GO, PUBG)、MOBA (王者荣耀) |

---

## 帧同步 (Lockstep)

### 原理
*   所有客户端在同一帧执行相同的操作指令。
*   如果所有客户端的初始状态一致，且每帧输入的指令一致，那么计算出的下一帧状态也必然一致。
*   服务器只负责转发操作指令，不进行逻辑计算（或仅做简单的校验）。

### 关键技术点
1.  **确定性 (Determinism)**：
    *   浮点数精度问题：不同 CPU/OS 对浮点数的处理可能微小差异，累积会导致不同步。
        *   解决方案：使用定点数 (Fixed Point Math) 库。
    *   随机数：必须使用确定性随机数生成器 (种子相同，序列相同)。
    *   容器遍历顺序：Map/Set 的遍历顺序可能因实现不同而不同。
2.  **追帧 (Catch-up)**：
    *   断线重连或加入游戏时，需要从头开始快速执行所有历史指令，恢复到当前状态。
    *   优化：定期保存**快照 (Snapshot)**，从最近的快照开始追帧。
3.  **预测与回滚 (Prediction & Rollback)** (高级帧同步，如 GGPO)：
    *   为了减少输入延迟感，客户端可以**预测**其他玩家的操作（通常预测为无操作或维持上一次操作）并先行计算。
    *   当收到服务器真实的指令后，如果与预测不符，则**回滚 (Rollback)** 到不符的那一帧，重新计算到当前帧。

## 状态同步 (State Sync)

### 原理
*   客户端上传操作（如“我按下了移动键”）。
*   服务器收到操作后，计算游戏逻辑（位置移动、伤害判定），并将结果（“玩家 A 在 (x,y) 位置，血量 100”）广播给周围的客户端。
*   客户端收到状态后，更新本地表现。

### 关键技术点
1.  **客户端预测 (Client-side Prediction)**：
    *   为了消除 RTT (Round Trip Time) 带来的操作延迟感（如按下移动键后 100ms 人物才动）。
    *   客户端按下按键后，**立即**在本地模拟移动，并表现出来。
    *   同时发送请求给服务器。
    *   收到服务器确认后，如果位置有偏差，进行平滑修正（**和解/Reconciliation**）。
2.  **插值 (Interpolation)**：
    *   **针对其他玩家** (Entity Interpolation)。
    *   由于网络抖动，收到其他玩家的位置更新是不均匀的（如 t1, t3, t4）。
    *   客户端不直接设置位置，而是将收到的状态放入缓冲区，**延迟**一段时间（如 100ms）进行渲染。
    *   在过去两个已知状态之间进行线性插值 (Lerp) 或平滑插值，使移动看起来平滑。
3.  **服务器回滚/延迟补偿 (Lag Compensation)** (常见于 FPS)：
    *   当玩家 A 开枪射击玩家 B 时，A 看到的是 B 在过去某个时刻的位置（因为 B 的位置传到 A 有延迟，A 看到的是插值后的 B）。
    *   服务器收到 A 的射击请求时，B 可能已经移动走了。
    *   服务器需要**回滚** B 的位置到 A 开枪的那个时刻，进行命中判定。

## 总结对比

*   **帧同步**：像“下棋”，每一步都得对齐，错了就全盘皆输。适合单位多、逻辑复杂但只需同步少量指令的游戏。
*   **状态同步**：像“看视频”，偶尔丢几帧也能看，服务器说了算。适合对安全性要求高、世界宏大、玩家中途加入频繁的游戏。
