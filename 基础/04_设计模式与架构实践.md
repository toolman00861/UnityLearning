# 04. 设计模式与 Unity 工程架构实践

在 Unity 开发中，良好的架构决定了项目的可维护性和扩展性。

## 1. 单例模式 (Singleton Pattern)

### 1.1 核心思想
保证一个类仅有一个实例，并提供一个全局访问点。
**Unity 实战**：常用于 `GameManager`, `AudioManager`, `InputManager`。

### 1.2 陷阱与优化
*   **MonoSingleton (继承 MonoBehaviour)**
    *   **问题**：场景切换时可能被销毁，再次访问会报错 `MissingReferenceException`。
    *   **解决**：使用 `DontDestroyOnLoad(gameObject)`。
    *   **问题**：多线程访问不安全。
    *   **解决**：使用双重检查锁定 (Double-Check Locking)。

```csharp
public class Singleton<T> : MonoBehaviour where T : MonoBehaviour {
    private static T _instance;
    private static object _lock = new object();

    public static T Instance {
        get {
            if (_instance == null) {
                lock (_lock) { // 线程安全
                    if (_instance == null) {
                        _instance = FindObjectOfType<T>();
                        if (_instance == null) {
                            GameObject go = new GameObject(typeof(T).Name);
                            _instance = go.AddComponent<T>();
                        }
                    }
                }
            }
            return _instance;
        }
    }

    protected virtual void Awake() {
        if (_instance == null) _instance = this as T;
        else if (_instance != this) Destroy(gameObject); // 保证唯一
        DontDestroyOnLoad(gameObject);
    }
}
```

## 2. 观察者模式 (Observer Pattern)

### 2.1 核心思想
定义对象间的一对多依赖关系，当一个对象改变状态，所有依赖者都会收到通知。
**Unity 实战**：`Event System`，`UI` 更新，`成就系统`。

### 2.2 实现方式
1.  **C# Event**：
    *   **优点**：简单，语言原生支持。
    *   **缺点**：强耦合（需要引用发布者实例），容易忘记退订导致内存泄漏。
2.  **消息中心 (Message Center / Event Bus)**：
    *   **优点**：完全解耦（发布者和订阅者都不认识对方，只认识 EventID）。
    *   **缺点**：类型不安全（参数通常是 object），难以追踪调用链。

```csharp
// 简易消息中心
public static class EventCenter {
    private static Dictionary<string, Action<object>> events = new Dictionary<string, Action<object>>();

    public static void AddListener(string name, Action<object> callback) {
        if (!events.ContainsKey(name)) events[name] = callback;
        else events[name] += callback;
    }

    public static void Broadcast(string name, object data) {
        if (events.TryGetValue(name, out var action)) action?.Invoke(data);
    }
    
    // 必须提供移除方法，否则 Delegate 会持有对象引用导致无法 GC！
    public static void RemoveListener(string name, Action<object> callback) { ... }
}
```

## 3. 对象池模式 (Object Pool Pattern)

### 3.1 核心思想
复用对象，避免频繁创建和销毁产生的 GC 和 CPU 开销。
**Unity 实战**：子弹、特效、怪物生成。

### 3.2 关键点
*   **借出 (Get)**：
    *   如果有空闲对象 -> 出栈/出队，激活 `SetActive(true)`。
    *   如果无空闲 -> `Instantiate` 新对象。
*   **归还 (Release)**：
    *   重置状态（位置、血量、特效），禁用 `SetActive(false)`，入栈/入队。
*   **脏数据处理**：归还时必须清理残留数据（如子弹速度、特效颜色），否则下次取出时会状态异常。

## 4. 状态机模式 (State Machine)

### 4.1 核心思想
让对象在其内部状态改变时改变其行为。
**Unity 实战**：角色控制（站立、跑、跳、攻击）、AI 行为（巡逻、追逐、攻击）。

### 4.2 实现方式
1.  **枚举 + Switch** (最简单，但逻辑复杂后难以维护)。
2.  **类状态机 (Class-based FSM)**：
    *   定义基类 `BaseState` (`Enter`, `Update`, `Exit`)。
    *   每个状态是一个类 `WalkState`, `JumpState`。
    *   上下文 `PlayerController` 持有当前状态引用。

```csharp
public abstract class State {
    protected Player player;
    public State(Player p) => player = p;
    public abstract void Enter();
    public abstract void Update();
    public abstract void Exit();
}

public class JumpState : State {
    public override void Enter() { player.PlayAnim("Jump"); }
    public override void Update() { 
        if (player.IsGrounded) player.ChangeState(new IdleState(player)); 
    }
    public override void Exit() { }
}
```

## 5. 命令模式 (Command Pattern)

### 5.1 核心思想
将请求封装为对象，从而使你可以用不同的请求对客户进行参数化。
**Unity 实战**：RTS 游戏操作、撤销/重做 (Undo/Redo) 系统、输入回放。

### 5.2 实现
*   `ICommand` 接口：`Execute()`, `Undo()`。
*   `CommandManager`：维护一个 `Stack<ICommand>` 历史记录。
*   **移动命令**：记录移动前的坐标和移动后的坐标。撤销时 `transform.position = oldPos`。

## 6. MVC / MVP / MVVM 架构

在 Unity UI 开发中，分离 逻辑 (Model) 和 表现 (View) 至关重要。

### 6.1 MVC (Model-View-Controller)
*   **Model**：数据 (PlayerHP)。
*   **View**：UI 组件 (Slider, Text)。
*   **Controller**：业务逻辑。
*   **缺点**：View 和 Model 往往耦合较重，Controller 容易臃肿。

### 6.2 MVVM (Model-View-ViewModel)
*   **核心**：**数据绑定 (Data Binding)**。
*   **ViewModel**：作为 View 和 Model 的中间人，暴露属性给 View 绑定。
*   **View**：监听 ViewModel 的属性变化（自动更新 UI）。
*   **Unity 方案**：虽然 uGUI 不原生支持，但可以使用 `UniRx` 或 `BindableProperty` 实现响应式绑定。

```csharp
// 简易 MVVM 绑定
public class IntProperty {
    public Action<int> OnValueChanged;
    private int _value;
    public int Value {
        get => _value;
        set { 
            if (_value != value) {
                _value = value;
                OnValueChanged?.Invoke(_value); // 通知 UI 更新
            }
        }
    }
}
```

## 面试自测题
1.  **单例模式在多线程下安全吗？**（不安全，需加锁 `lock` 或使用 `Lazy<T>`）
2.  **观察者模式最大的坑是什么？**（内存泄漏：忘记移除监听导致对象无法被 GC）
3.  **状态机如何处理状态间的切换条件？**（可以在 State 的 Update 中判断，或者由外部 Manager 统一调度 Transition）
4.  **MVC 和 MVVM 的最大区别？**（MVVM 引入了数据绑定，View 和 ViewModel 之间是自动同步的，无需手动 `text.text = model.name`）
