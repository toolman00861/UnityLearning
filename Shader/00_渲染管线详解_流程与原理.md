# 渲染管线详解 (The Rendering Pipeline)

渲染管线（Rendering Pipeline）是图形学中最核心的概念，它描述了**从 3D 场景数据到最终屏幕上 2D 图像**的完整生成过程。

你可以把它想象成工厂的**流水线**：
*   **原材料**: 模型、贴图、灯光、摄像机数据。
*   **产品**: 屏幕上显示的图像。
*   **流水线**: CPU 和 GPU 协同工作的各个阶段。

通常，我们将渲染管线划分为三个主要阶段：
1.  **应用阶段 (Application Stage)**：由 CPU 负责。
2.  **几何阶段 (Geometry Stage)**：由 GPU 负责。
3.  **光栅化阶段 (Rasterizer Stage)**：由 GPU 负责。

---

## 1. 应用阶段 (Application Stage) - CPU 主导

在这个阶段，开发者拥有绝对的控制权。主要任务是准备好“要画什么”和“怎么画”。

### 1.1 准备场景数据
CPU 需要将显卡需要的数据加载到显存中：
*   **模型数据**: 顶点位置、UV、法线、切线等。
*   **渲染状态**: 材质参数、Shader、纹理贴图、混合模式等。
*   **变换矩阵**: 告诉 GPU 物体在哪里（位置、旋转、缩放）。

### 1.2 粗粒度剔除 (Culling)
为了提高性能，CPU 会把那些**完全不在摄像机视野内**的物体剔除掉，不交给 GPU 处理。
*   **视锥体剔除 (Frustum Culling)**: 最常用的技术。

### 1.3 设置渲染状态与 Draw Call (重要!)
*   **SetPass Call**: CPU 告诉 GPU：“接下来我要用这个材质（Shader + 纹理）来画画了”。这涉及到切换显卡的渲染状态，开销较大。
*   **Draw Call**: CPU 告诉 GPU：“数据准备好了，按照刚才的设置，把这个模型画出来！”。
    *   **性能瓶颈**: Draw Call 太多会导致 CPU 忙于发号施令，而 GPU 经常闲置等待。
    *   **优化**: **批处理 (Batching)** 就是把多个小模型合并成一个大模型，用一次 Draw Call 画完。

---

## 2. 几何阶段 (Geometry Stage) - GPU 处理顶点

一旦 CPU 发出 Draw Call，GPU 就开始接手。这个阶段主要处理**顶点 (Vertex)** 数据。

### 2.1 顶点着色器 (Vertex Shader) - 【完全可编程】
这是渲染管线中第一个我们可以写代码控制的阶段。
*   **核心任务**: **坐标变换**。
    *   将顶点坐标从 **模型空间 (Local Space)** -> **世界空间 (World Space)** -> **观察空间 (View Space)** -> **裁剪空间 (Clip Space)**。
    *   也就是大名鼎鼎的 **MVP 变换**。
*   **次要任务**: 逐顶点光照（效果较差，现在少用）、准备传给片元着色器的数据。

### 2.2 裁剪 (Clipping) - 【硬件固定】
将那些**部分在视野内、部分在视野外**的图元（如三角形）进行裁剪。
*   完全在视野外的被丢弃。
*   跨越边界的会被“剪断”，生成新的顶点，确保所有数据都在裁剪空间内。

### 2.3 屏幕映射 (Screen Mapping) - 【硬件固定】
将裁剪空间下的 3D 坐标（范围 -1 到 1）转换到 **屏幕像素坐标**（例如 1920x1080）。
*   注意：这一步保留了 **Z 坐标（深度值）**，用于后续的深度测试。

---

## 3. 光栅化阶段 (Rasterizer Stage) - GPU 生成像素

这个阶段的任务是计算：**屏幕上的哪些像素被这个三角形覆盖了？这些像素应该是什么颜色？**

### 3.1 三角形遍历 (Triangle Traversal)
*   检查每个像素是否被三角形网格覆盖。
*   如果覆盖，就生成一个 **片元 (Fragment)**。
*   **插值 (Interpolation)**: 根据三角形三个顶点的信息，计算出内部每个片元的信息（如 UV 坐标、法线方向）。

### 3.2 片元着色器 (Fragment Shader) - 【完全可编程】
又称像素着色器 (Pixel Shader)。这是最为耗时、效果最关键的阶段。
*   **输入**: 经过插值的片元数据。
*   **任务**: 计算该像素的最终颜色。
    *   **纹理采样**: 根据 UV 坐标去图片上取颜色。
    *   **光照计算**: 结合法线、光照方向、视线方向，计算漫反射、高光、阴影。
*   **输出**: 一个颜色值 (RGBA)。

### 3.3 逐片元操作 (Per-Fragment Operations) - 【高度可配置】
也称为 **输出合并 (Output Merger)** 阶段。这里决定了片元“能不能”显示在屏幕上，以及“如何”显示。

一系列测试流水线（**谁没通过谁就死**）：
1.  **模板测试 (Stencil Test)**:
    *   与模板缓冲区 (Stencil Buffer) 比较。常用于实现遮罩效果（如传送门、魔窗）。
2.  **深度测试 (Depth Test)**:
    *   **Z-Buffer (深度缓冲)**: 存储屏幕上每个像素当前的深度值（离相机多远）。
    *   **逻辑**: 如果新片元比缓冲区里的值**更近**，说明它挡住了后面的物体 -> **通过测试**，更新 Z-Buffer。否则 -> **丢弃**（被遮挡了，看不见）。
    *   **Early-Z**: 现代 GPU 为了优化，通常会把深度测试提前到片元着色器之前执行（如果片元着色器不修改深度值的话），避免计算了半天最后被遮挡看不见。

3.  **混合 (Blending)**:
    *   如果片元通过了所有测试，且物体是**半透明**的，就需要与颜色缓冲区中已有的颜色进行混合（Alpha Blending）。
    *   公式：`最终颜色 = 源颜色 * 源因子 + 目标颜色 * 目标因子`。

## 4. 总结图解

| 阶段 | 子流程 | 主导者 | 可编程性 | 核心任务 |
| :--- | :--- | :--- | :--- | :--- |
| **应用阶段** | 准备数据、剔除、DrawCall | **CPU** | 逻辑完全控制 | 告诉 GPU 画什么 |
| **几何阶段** | **顶点着色器** | GPU | **Shader (Vertex)** | 坐标变换 (MVP) |
| | 裁剪、屏幕映射 | GPU | 硬件固定 | 3D 转 2D 屏幕坐标 |
| **光栅化阶段** | 三角形遍历 | GPU | 硬件固定 | 找像素、插值 |
| | **片元着色器** | GPU | **Shader (Fragment)** | 算颜色、采样、光照 |
| | 深度/模板测试、混合 | GPU | 配置 (ZTest/Blend) | 遮挡关系、透明混合 |

### 常见面试题映射
1.  **为什么 DrawCall 多了会卡？** -> CPU 在应用阶段忙于切换状态和提交命令，GPU 没吃饱。
2.  **Vertex Shader 和 Fragment Shader 谁执行次数多？** -> 通常 Fragment 多得多（一个三角形3个顶点，但可能覆盖几百个像素）。所以复杂的计算尽量放 VS 里（如果效果允许）。
3.  **透明物体为什么要最后渲染？** -> 因为透明物体需要和背景混合（Blending）。如果先画透明物体并写入深度，后面的不透明物体会被深度测试剔除，导致透明物体后面是空的；如果不写深度，又无法正确遮挡。标准做法：**不透明物体从近到远(写深度) -> 透明物体从远到近(只读深度，开启混合)**。
